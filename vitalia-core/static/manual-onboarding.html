Bem-vindo ao Vitalia Agente Local! Assume conhecimento geral de Python e terminal Linux, mas não assume familiaridade com AutoGen ou Redis Streams.

💡
Audiência: Colaborador técnico novo ao projeto. Leia as seções na ordem — cada uma é pré-requisito da próxima.
1

O que é o Vitalia?

O Vitalia é um orquestrador de agentes de IA que coordena dois modelos de linguagem (LLMs) rodando em máquinas distintas para executar tarefas de desenvolvimento de software de forma colaborativa.

Pense assim: você dá uma tarefa ("implemente uma função Python para X"). Dois agentes entram em ação:

  • O Arquiteto — pensa, pesquisa na web, toma decisões de design.
  • O Engenheiro — escreve o código, salva no banco de dados e reporta o que foi feito.

Eles se comunicam em turnos (Round-Robin), mediados pelo framework AutoGen da Microsoft.

ℹ️
O que é AutoGen? É um framework Python que orquestra conversas entre múltiplos agentes de IA. Cada agente tem um modelo de linguagem associado, um conjunto de ferramentas (funções Python que pode chamar) e um papel definido. O AutoGen gerencia os turnos, o histórico de mensagens e a condição de parada.
2

Arquitetura em Camadas

2.1 Duas Máquinas, Um Sistema

Nó 1 — Notebook (você)          Nó 2 — Servidor (GTX 1060)
─────────────────────           ─────────────────────────────
• Arquiteto (llama3.2:3b)       • Engenheiro (qwen2.5-coder:7b)
• Redis (vitalia_redis)         • Inferência GPU
• pgvector (vitalia_db)
• Dashboard (telemetry_api)
• Open WebUI
💡
Por que duas máquinas? Modelos de linguagem grandes exigem muita memória de vídeo (VRAM). O servidor tem uma GPU dedicada (GTX 1060 com 6GB de VRAM) que roda o modelo de código mais poderoso. O notebook roda o modelo mais leve em CPU. Juntos, fazem mais do que qualquer máquina sozinha conseguiria.

2.2 O Problema do Tool Calling e o Tool Bridge

Ferramentas são funções Python que o LLM pode "chamar" (ex: pesquisar na web, salvar código no banco). Mas o llama3.2:3b rodando em CPU no Nó 1 não sabe chamar ferramentas no formato correto nativamente. Em vez disso, ele emite JSON em texto puro.

O Tool Bridge intercepta esse JSON e usa o Redis Streams para orquestrar a execução:

LLM no Nó 1 responde:                  O sistema intercepta:
────────────────────────                ─────────────────────────────
{"name": "web_search",       →   VitaliaOllamaClient detecta o JSON
 "arguments": {"query": "x"}}    └─► posta pedido no Redis Stream
                                       └─► tool_worker() executa a função
                                            └─► posta resultado no Redis Stream
                                                 └─► AutoGen recebe como se fosse
                                                     uma chamada de ferramenta normal
ℹ️
O que é Redis Streams? Redis é um banco de dados em memória ultra-rápido. Streams são como filas de mensagens. Cada mensagem tem um ID único. No Vitalia, usamos duas streams por agente: tool_requests e tool_results. A comunicação é assíncrona.

2.3 Containers Docker

O sistema usa 4 containers gerenciados pelo Docker Compose:

Container Porta Papel
vitalia_redis 6379 Fila de mensagens + cache de contexto
vitalia_db 5432 Banco pgvector (memória de código/RAG)
vitalia_ollama 11434 Motor de inferência local (Nó 1)
vitalia_open_webui 3000 Interface web para chat direto com os modelos
ℹ️
O que é pgvector? É uma extensão do PostgreSQL que armazena e busca vetores numéricos — representações matemáticas do significado de texto. Quando o Engenheiro salva código, o texto é convertido em vetor. Nas próximas sessões, o sistema busca código similar por semântica.

Pré-requisitos e Instalação

O que você precisa ter instalado

  • Docker + Docker Compose — para rodar os containers
  • Python 3.12+ — para o orquestrador (recomendado usar .venv)
  • Acesso de rede ao Nó 2 (servidor) na porta 11434

Pipeline de Validação E2E

O teste E2E é dividido em 6 fases de sanidade (A-F). Como o processo do zero até a inferência envolve muita coisa (Docker, Rede, Python, Tool Bridge, Modelos), se a inferência falhar é inútil debugar diretamente nela sem garantir a fundação antes. Esta é a rotina para validar se o ambiente subiu 100%.

Infraestrutura
Confirmar que os 4 containers Docker estão operacionais
bash — A.0 Containers
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
bash — A.1 Redis
docker exec -it vitalia_redis redis-cli -a vitalia_redis_secure_2026 ping
ℹ️
O warning "Using a password with '-a'…" é esperado e inofensivo no Redis CLI. O que confirma o sucesso é receber o PONG.

Solução de Problemas Comuns

🔴 "Connection refused" ao acessar o Ollama do Nó 2 (servidor) ▼
Causa

O Ollama no servidor está configurado para escutar apenas localmente (127.0.0.1), não aceitando conexões de outras máquinas da rede.

Solução

No servidor, defina a variável de ambiente OLLAMA_HOST=0.0.0.0 antes de iniciar o processo do Ollama ou adicione essa variável no systemd service/docker-compose dele.

🔴 Tool call aparece como texto normal na resposta do LLM ▼
Causa

O llama3.2:3b (ou o modelo ativo no Nó 1) optou por falar em texto em vez de gerar o JSON exato exigido pelo prompt do sistema.

Solução

Isso não é um bug sistêmico, e sim comportamental. Para garantir a formatação, re-execute, ou edite o system_message com exemplos claros de JSON. Caso aconteça, o AutoGen tentará intervir ou o Bridge passará em branco interpretando como texto.

🔴 Worker timeout após 30s chamando ferramentas ▼
Causa

O worker de background que processa o Redis Streams não subiu. Se você colocar as duas variáveis TOOL_CALLING_NATIVE como true no .env, o orquestrador assume que não precisa subir o worker.

Solução

Garantir que pelo menos uma das variáveis TOOL_CALLING_NATIVE esteja false para que o tool_worker do Tool Bridge inicie junto com o run_vitalia().

🔴 Memória cheia (OOM) / lentidão absurda no Nó 2 ▼
Causa

A GPU (GTX 1060 de 6GB) lotou a memória de vídeo (VRAM). O qwen2.5-coder:7b já ocupa cerca de 5.5GB só para pesos e janelas de contexto restritas.

Solução

Reinicie o Ollama no servidor para liberar a VRAM retida em cache (docker restart vitalia_ollama ou sudo systemctl restart ollama).
