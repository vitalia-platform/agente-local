Decisões de Design (ADRs)

ADR-01 Tool Bridge via Redis Streams (não via callback direto) ▼
Problema

O AutoGen roteia a resposta do LLM antes que possamos injetar o resultado da ferramenta no mesmo turno.

Decisão

O wrapper VitaliaOllamaClient.create() detecta o JSON, posta em vitalia:tool_requests:<agent>, aguarda com asyncio.Event e retorna FunctionCall ao AutoGen — como se o tool call tivesse sido nativo.

Trade-off aceito

Latência de 1 round-trip Redis (~1ms local) por tool call. Aceitável dado que a inferência já leva segundos.

Alternativa rejeitada

Monkey-patch no AutoGen — frágil a atualizações de versão.

ADR-02 correlation_id (UUID4) por pedido de ferramenta ▼
Problema

Múltiplos tool calls em paralelo poderiam cruzar respostas na stream.

Decisão

Cada pedido gera um UUID4. O tool_worker() posta o resultado com o mesmo correlation_id. O wrapper filtra por ele no XREAD.

Armadilha conhecida

Se o worker crashar após o XADD de request mas antes do XADD de result, o wrapper ficará bloqueado até TOOL_BRIDGE_TIMEOUT_SEC. O resultado é uma mensagem de erro descritiva ao LLM — sem crash do sistema.

ADR-03 HeadAndTailChatCompletionContext no Engenheiro ▼
Problema

O qwen2.5-coder:7b com 6GB de VRAM não aguenta contexto longo.

Decisão

Com head_size=1 e tail_size=20, o Engenheiro sempre vê a tarefa inicial (head) e as últimas 20 mensagens (tail).

Armadilha

Se o Engenheiro estiver repetindo respostas, verifique se tail_size=20 é suficiente. Aumente para 30 se necessário, mas monitore a VRAM.

ADR-04 Sem retry automático no Tool Bridge ▼
Decisão

Timeout → erro descritivo legível pelo LLM, que pode decidir tentar novamente ou reportar o problema.

Rationale

Retry automático mascararia falhas sistêmicas (Redis down, worker crash). O LLM é o agente correto para decidir se deve ou não tentar de novo.

import asyncio, os
from dotenv import load_dotenv
load_dotenv("../.env")
import redis.asyncio as redis_async

REDIS_URL = f"redis://:{os.getenv('REDIS_PASSWORD')}@localhost:{os.getenv('REDIS_PORT', 6379)}/0"

async def test():
    r = redis_async.Redis.from_url(REDIS_URL)
    cid = "bench-test-001"
    req = "vitalia:tool_requests:Architect"
    res = "vitalia:tool_results:Architect"
    await r.delete(req, res)
    await r.xadd(req, {"correlation_id": cid, "tool_name": "web_search",
        "arguments_json": '{"query":"test"}', "agent_name": "Architect",
        "timestamp": "2026-06-30T20:00:00"})
    print(f"✅ XADD {req}")
    await r.xadd(res, {"correlation_id": cid, "result": "OK", "error": "", "duration_ms": "1"})
    print(f"✅ XADD {res}")
    msgs = await r.xread(streams={res: "0"}, count=10, block=1000)
    found = any(f.get(b"correlation_id", b"").decode() == cid
                for _, ms in msgs for _, f in ms)
    print("✅ XREAD OK" if found else "❌ Não encontrado")
    await r.aclose()
    print("✅ Tool Bridge Channel: PASS")

asyncio.run(test())
EOF

Troubleshooting & Diagnóstico

🔴 NO1_MODEL: None ao rodar a Fase C ▼
Causa

A variável NO1_MODEL (ou NO2_MODEL) não está no .env. O os.getenv() retorna None e não falha na inicialização, só na inferência.

Solução

Adicionar ao .env: NO1_MODEL=llama3.2:3b e NO2_MODEL=qwen2.5-coder:7b.

🔴 Tool call aparece como TextMessage (não ToolCallRequestEvent) ▼
Causa

O LLM não gerou o JSON {name, arguments} esperado. Pode ter ignorado o system_message ou não entendido a instrução de formato.

Solução

Verificar nos logs do Redis se vitalia:tool_requests:Architect tem entradas. Se não tiver, o LLM não gerou JSON — o sistema tratou como texto. Reforçar o system_message com exemplo explícito do formato JSON.

🔴 Worker timeout — "Tool unavailable: worker timeout after 30s" ▼
Causa

O tool_worker() não foi lançado. Isso acontece quando ambos NO1_TOOL_CALLING_NATIVE=true e NO2_TOOL_CALLING_NATIVE=true — o run_vitalia() não lança o worker neste caso.

Solução

Confirmar com grep "TOOL_CALLING_NATIVE" .env. Se ambos são true, o Bridge não é necessário — mas verifique se deveria ser false em algum nó.

🔴 VRAM cheia no Nó 2 — inferência trava ou retorna OOM ▼
Causa

O qwen2.5-coder:7b ocupa ~5.5GB dos 6GB disponíveis. Outro processo pode estar usando a GPU.

Solução

No servidor: nvidia-smi para identificar processos. Reiniciar o Ollama libera a VRAM: docker restart vitalia_ollama.

🔴 ConnectionRefusedError ao pingar o Ollama do Nó 2 (Fase B.2) ▼
Causa

O Ollama no servidor está escutando só em 127.0.0.1 (loopback), não na interface de rede.

Solução

No servidor: ss -tlnp | grep 11434. Se retornar 127.0.0.1, configurar OLLAMA_HOST=0.0.0.0 antes de iniciar o Ollama.

🔴 Dashboard JWT expirado ou não autentica ▼
Causa

Token JWT de 60 minutos expirou, ou o DASHBOARD_SECRET_KEY mudou no .env após o login.

Solução

Logout e re-autenticar. Se persistir, reiniciar a API: uvicorn telemetry_api:app --host 0.0.0.0 --port 8000 --reload
